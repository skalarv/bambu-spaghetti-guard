Spaghetti Guard — Installation & Operation Manual

A self-hosted print-failure detector. It watches the P1S chamber camera at ~1 fps, runs a YOLO model on every frame, and pauses or stops the print over MQTT when a spaghetti or detachment failure is confirmed across several consecutive frames.

Overview

The guard connects to the printer's LAN interface on two channels: the camera stream (TCP 6000, TLS) and the MQTT report/command bus (TCP 8883, TLS). It only runs detection while the printer reports it is actively printing (gcode_state == RUNNING), and it never publishes a stop on the mere absence of camera frames. On a confirmed failure it takes the configured action (pause / stop / ask), writes a snapshot JPEG, and pushes a notification.

Status — Detection validated on a real ~2.4 hour print with zero false triggers. ntfy notifications are live and delivering to a subscribed phone.

Installation

1 Prepare the printer

  1. On the P1S touchscreen: Settings → General → LAN Mode (or Developer Mode on older firmware) → turn it on.
  2. In your router, give the P1S a static IP (DHCP reservation by MAC).
  3. Note the LAN access code (Settings → General → LAN Mode) and the serial number (Settings → Device).
Note The guard refuses to publish if LAN Mode is off, and refuses to connect if the printer IP isn't reachable.

2 Set up the workstation

git clone https://github.com/skalarv/bambu-spaghetti-guard.git
cd bambu-spaghetti-guard
.\tasks.ps1 setup          # venv + pinned deps + editable install

For the live detector, add CUDA-PyTorch and Ultralytics. Use the CUDA index matching your GPU (cu128 for an RTX 50-series / Blackwell card):

.\.venv\Scripts\Activate.ps1
pip install torch --index-url https://download.pytorch.org/whl/cu128
pip install -e .[live]
Why separate torch/ultralytics/opencv are intentionally not default deps so the test install stays fast and disk-light.

3 Configure secrets

copy secrets.local.txt.template secrets.local.txt

Edit secrets.local.txt (git-ignored — never committed):

BAMBU_IP=192.168.50.51
BAMBU_SERIAL=01P00A3B1300712
BAMBU_ACCESS_CODE=<8-char code from touchscreen>

Both run and live-verify load this file automatically; explicit environment variables take precedence. The access code is env-only — putting it in config.yaml fails validation. Then tune config.yaml: action.mode (start with pause), detector.conf_threshold, and detector.consecutive_hits.

4 Push notifications (ntfy)

The quickest path, no signup:

  1. Pick a unique, hard-to-guess topic (ntfy topics are public), e.g. spaghetti-guard-yz-cc3d60022c, and set it in secrets.local.txt: 
    NTFY_TOPIC_URL=https://ntfy.sh/<your-topic>
  2. Set the backend in config.yaml: 
    notify:
  backend: ntfy
  3. Install the ntfy app on your phone and subscribe to the same topic. Until you subscribe, alerts reach the server but not your phone.
TLS on inspected networks The notifier verifies HTTPS against the OS trust store (ssl.create_default_context() in notifier.py), not the bundled certifi roots. On a corporate / TLS-inspection LAN this is what lets alerts through. Do not "fix" a CERTIFICATE_VERIFY_FAILED by installing pip-system-certs — it globally patches ssl and breaks the P1S MQTT handshake.

5 Get a model

The active weights ship at models/yolo11n-spaghetti.pt (yolo26n fine-tuned on 24,098 CC-BY images; val P=0.86 R=0.68 mAP50=0.72). Point detector.model_path at your own weights to swap it. See docs/RETRAINING.md to adapt the model to your chamber.

6 Verify offline (mandatory)

.\tasks.ps1 test                       # full unit + integration suite
.\tasks.ps1 replay <path-to-clip>       # tune thresholds on recorded footage

Every test must pass before pointing the guard at the real machine.

7 Dry-run against a real print

.\tasks.ps1 run-dry

This runs the full live pipeline — real camera, real detector, real MQTT parsing — but logs the action payload instead of publishing it. Run through a normal print (confirm it stays silent) and a deliberately failed one (an already-detached print is the easiest reproducer). You'll see guard ARMED when the print starts and any Print-failure detected lines it would have acted on.

Safe end-to-end check You can also run spaghetti-guard live-verify — dry-run + a never-fires detector + the viewer window — purely to confirm the camera handshake on new firmware.

8 Go live

When you trust the detection, run for real (drops --dry-run):

.\tasks.ps1 run-live            # starts in action.mode = pause

Or double-click the desktop launcher Spaghetti Guard - LIVE.bat, which runs spaghetti-guard run --viewer (real publish) and keeps the camera window open. Ctrl+C shuts the guard down cleanly; the printer keeps printing.

Live means live In this mode a confirmed failure sends a real pause/stop to the printer. Promote from pause to stop only after you've watched several real triggers behave correctly.

Operation

Daily check (15 seconds)

  1. Is the guard process alive? (Get-Process spaghetti-guard, or the LIVE.bat window is still open.)
  2. Is the most recent log line within the last minute?
  3. Is failure_snapshots/ empty since your last review? If not, review the JPEGs.

Action modes

ModeBehaviourUse when
pausePauses the print; hot-end stays warm; filament saved for inspection.Building trust; a person is nearby to act.
stopStops the print immediately; frees the printer; the print is discarded.Long unattended prints, once detection is trusted.
askShows a confirm popup in the viewer; on timeout falls back to ask_timeout_action.You watch the viewer and want the final call.

Alert events

With notify.backend: ntfy, these push to your phone:

AlertTrigger
Spaghetti detectedConfirmed failure (N consecutive hits). Includes the snapshot image.
Action FAILEDA pause/stop command did not reach the printer — check it immediately. Retries every ~10 s.
Camera silentNo frame for camera.timeout_s while armed. Logged; no action taken.
Printer reports silentNo MQTT report for ~30 s — the guard may be silently unprotected.

Troubleshooting

SymptomDiagnosisFix
MQTT command verification failedLAN Mode disabled or wrong access code.Re-enable LAN Mode; re-copy the access code into secrets.local.txt.
socket.gaierror on startupBAMBU_IP wrong or printer offline.Confirm the static IP; ping the printer.
Trips during normal printsThreshold/debounce too low for this chamber.Raise conf_threshold +0.05 or consecutive_hits +1–2; retrain on the false-positive footage.
Every alert CERTIFICATE_VERIFY_FAILEDTLS-inspection proxy root CA not in certifi.Handled — notifier uses the OS trust store. Do not install pip-system-certs.
No ntfy push (log says delivered)Not subscribed on the phone.Open the ntfy app → subscribe to the exact NTFY_TOPIC_URL topic.
Camera-loss alert keeps firingStream slower than camera.timeout_s.Raise camera.timeout_s (e.g. 15 → 20); confirm the printer IP is stable.

Safety posture

  • This is a last line of defence against runaway spaghetti — it is not a fire alarm. It cannot see smoke, hot-end fire, or thermal runaway.
  • Detection runs only while the printer reports RUNNING; it disarms on FINISH / FAILED / IDLE / PAUSE.
  • Camera loss never publishes a stop — a missing frame is not evidence of failure.
  • The pause/stop publish is verified against a broker ack; an unconfirmed command keeps the guard TRIGGERED, alerts you, and retries rather than pretending it acted.
  • For long unattended prints prefer stop over pause — a paused printer with a hot nozzle is marginally worse than a stopped one.
  • To stop the printer manually, use the touchscreen or Bambu app — not by killing the guard.
