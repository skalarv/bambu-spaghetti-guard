# Failure-detection datasets

Goal: a YOLO-format dataset that fine-tunes well to the P1S stock chamber
camera.

## Canonical class taxonomy (post-merge)

Since 2026-07-06 the training pipeline uses a 7-class canonical taxonomy that
aligns the noisy per-source class names into a single unified vocabulary:

| Class id | Name | Notes |
|---|---|---|
| 0 | `spaghetti` | Tangled extrusion above the build plate. Dominant failure to catch. |
| 1 | `stringing` | Thin filament strands between features. Usually cosmetic. |
| 2 | `blob` | Localised over-extrusion / warping blob on the print top. |
| 3 | `crack` | Layer separation / vertical fissure on the side wall. |
| 4 | `detachment` | Print lifted from the bed; nozzle dragging a moving part. |
| 5 | `over_extrusion` | Extrusion mass exceeds target — bulging profile. |
| 6 | `under_extrusion` | Extrusion mass below target — gappy or missing layers. |

The **guard's `failure_classes` in `config.yaml`** is a subset — currently
`[spaghetti, detachment]` — because these are the two failures that warrant
stopping a print. The other five are still detected and reported to the viewer
but do not fire the state machine. Tune per operator preference.

## Merged training set

`training/data/merged/` is built by `training/merge_datasets.py` from three
CC-BY 4.0 Roboflow Universe datasets:

| Source (slug) | Roboflow URL | Images | Classes | Notes |
|---|---|---|---|---|
| `s3d` — spaghetti-3d | `3d-printing-failure/spaghetti-3d` v1 | 715 | 2 clean | `spaghetti`, `stringing` |
| `flw` — 3d-printing-flaws | `spaghettidetect/3d-printing-flaws` v3 | 9,478 | 9 noisy → 5 canonical | Typo-remap: `Spagatti`→`spaghetti`, `poor initial layer bed adhesion faiure`→`detachment`, `fail`→`blob`, etc. |
| `syl` — sylucauc-3dpfd | `sylucauc/3d-printing-failure-detection` v3 | 13,905 | 6 clean | `blobs`, `cracks`, `over_extrusion`, `spaghetti`, `stringing`, `under_extrusion` |

Total: **24,098 images, 101,984 annotated boxes** across 7 canonical classes.

### Reproducing the merge

```powershell
# 1. Get a free Roboflow API key from roboflow.com/settings/api and add it to
#    secrets.local.txt as ROBOFLOW_API_KEY=<your-key>
# 2. Windows behind a TLS-inspecting router/proxy? Install pip-system-certs
#    to route SSL through the Windows cert store. IMPORTANT: uninstall it
#    before running the guard — it breaks paho-mqtt's CERT_NONE handshake:
pip install pip-system-certs        # for Roboflow downloads only
python training\download_datasets.py  # or the ad-hoc snippet in bank/
pip uninstall -y pip-system-certs   # BEFORE running the guard
# 3. Merge
python training\merge_datasets.py
```

## Layout on disk

Each source under `training/data/<source>/` and the merged output at
`training/data/merged/` follow the Ultralytics YOLOv8 layout Roboflow exports
by default:

```
training/data/merged/
├── data.yaml       # generated by merge_datasets.py — absolute paths
├── LICENSE.txt     # CC BY 4.0 attribution
├── SOURCES.md      # per-source provenance
├── train/
│   ├── images/     # *.jpg (source-slug-prefixed to avoid collisions)
│   └── labels/     # *.txt (YOLO format, remapped class ids)
├── valid/
│   ├── images/
│   └── labels/
└── test/
    ├── images/
    └── labels/
```

## Operator capture protocol (chamber-specific data)

Chamber-specific footage beats public data long-term. The `failure_snapshots/`
directory accumulates a `trigger-YYYYMMDD-HHMMSS-<class>-<conf>.jpg` for every
guard fire. See `docs/RETRAINING.md` for the review + promote + retrain cycle.

Rules of thumb:

1. Same camera, same chamber lighting as production.
2. Capture at the operating fps (~1 fps).
3. For each captured print, log: filament, layer height, model name, outcome
   (success / failure-type / aborted).
4. Annotate failure frames with the **failure start frame**, not just bounding
   boxes — the metrics tool needs onset for latency.
5. Always keep an untouched copy of the raw clip; the labels evolve as the
   taxonomy is refined.
6. **Save empty `.txt` files for false positives** — they train the model to
   shut up on look-alikes.

## Provenance and license tracking

Every dataset folder carries a `LICENSE.txt` and a `SOURCES.md` describing
where the images came from and what license applies. The training pipeline
(`training/prepare_dataset.py`) refuses to build a `data.yaml` if either is
missing (override with `--ignore-provenance`, but not for shipped models).

Weights derived from CC BY 4.0 sources require attribution when
redistributed. `models/yolo11n-spaghetti.pt` is derived from all three CC BY
4.0 datasets above.

## Excluded sources

Per brief §1: no Chinese-origin models or weights. Only Ultralytics-published
base weights (`yolo11n.pt`, `yolo11s.pt`, `yolo26n.pt`) or self-trained
derivatives.
